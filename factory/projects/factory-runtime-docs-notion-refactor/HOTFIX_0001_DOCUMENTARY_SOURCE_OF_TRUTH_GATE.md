# HOTFIX-0001 — Documentary Source-of-Truth Gate

Project: `factory-runtime-docs-notion-refactor`
Branch: `factory/runtime-docs-notion-refactor/hotfix-doc-source-truth-gate`
Opened: 2026-06-10
Status: OPEN — planning contract created, implementation pending Factory worker execution

## Jean directive

Open a hotfix in the same Factory project to correct the Factory methodology/runtime:

- Notion is important for project visibility, executive reporting, user-facing/director-facing status, and the project register.
- Notion is **not** the source of truth.
- Canonical source of truth is the engineering documentation committed in the repository plus Agent Core Factory DB state/gates/events.
- The first Factory gate must include a required-document checklist; all blocking required documents must be created, validated, reviewed, indexed, and committed before implementation can advance.
- A project starts from a complete engineering-level documentary source of truth: requirements analysis, pattern analysis, hypothesis/assumption analysis, technical blueprint, scope clarification, and planning.
- During execution, additional documents may be generated by responsible actors (reviews, change records, sprint close reports, retrospectives, Notion updates), but the blocking required documents must already exist before implementation.

## Scope

This hotfix corrects the Factory control-plane contract and runtime checks so non-trivial Factory projects follow this order:

1. G0 Repository Strategy Gate.
2. G1 Documentary Readiness Gate.
3. Implementation / review / QA / security / delivery gates.

The hotfix is limited to Factory runtime/methodology. It does **not** unfreeze or refactor CRM/Funnel Core.

## Canonical source-of-truth model

| Layer | Purpose | Blocking source of truth? |
|---|---|---|
| Agent Core Factory DB | Operational state: projects, lanes, tasks, gates, events, runs, anomalies | Yes |
| Versioned repo Markdown docs | Engineering contract: requirements, scope, architecture, planning, gates, reviews | Yes |
| Git commits/worktrees | Evidence checkpoints and immutable history | Yes |
| Notion PM page | Human-readable projection, executive report, project register, quick links, sprint narrative | No, unless Jean explicitly makes it mandatory for a specific project |
| Wiki/skills/session memory | Agent knowledge/procedures/history | No for project acceptance |

## Required documents before implementation

The G1 Documentary Readiness Gate must verify these documents for every non-trivial Factory project before normal implementation dispatch:

| Document | Purpose | Must exist before implementation? | Must be reviewed before implementation? |
|---|---|---:|---:|
| `FACTORY_INTAKE.md` | Trigger, requester, context, boundaries, stakeholders, non-goals | Yes | Yes |
| `REQUIREMENTS_ANALYSIS.md` | Detailed functional/business requirements, ambiguity log, user clarifications needed | Yes | Yes |
| `PATTERN_ANALYSIS.md` | Existing repo/runtime patterns, prior comparable implementations, anti-patterns to avoid | Yes | Yes |
| `ASSUMPTIONS_AND_OPEN_QUESTIONS.md` | Hypotheses, assumptions, risks, explicit open questions and resolution policy | Yes | Yes |
| `PRD.md` | Product requirements, acceptance criteria, success metrics | Yes | Yes |
| `ADRS.md` | Architecture decisions and trade-offs | Yes | Yes |
| `METHODOLOGY_PLAN.md` | Factory methodology, autonomy, gate order, roles, reviewer independence | Yes | Yes |
| `TECHNICAL_BLUEPRINT.md` | Components, modules, data contracts, integration boundaries, testing strategy | Yes | Yes |
| `SPRINT_PLAN.md` | Sprint/increment plan, milestones, sequencing | Yes | Yes |
| `TASK_GRAPH.md` | Task graph/story map/dependencies/owners/reviewers/evidence | Yes | Yes |
| `TRACKER.md` | Project-local status tracker derived from Factory DB and repo artifacts | Yes | Yes |
| `QA_GATES.md` | QA/check criteria before delivery | Yes | Yes |
| `SECURITY_GATES.md` | Security/privacy/secrets/payment/auth gates when applicable; otherwise explicit N/A rationale | Yes | Yes |
| `DOCUMENTATION_INDEX.md` | Canonical document hub listing required docs, optional docs, status, links, owner/reviewer | Yes | Yes |

## Lifecycle documents generated during execution

These are important but may be created/updated during the project by the responsible actors:

| Document | Owner | Timing |
|---|---|---|
| `QA_REPORT.md` | QA verifier | During/after QA gate |
| `SECURITY_REVIEW.md` | Security reviewer | During/after security gate |
| `QUALITY_REVIEW.md` | Independent quality reviewer | During review gate |
| `DELIVERY_REPORT.md` | Factory reporter/orchestrator | Delivery gate |
| `NOTION_UPDATE.md` | Factory reporter | Sprint planning/close/project close, when Notion projection is enabled |
| `CHANGELOG.md` or `CHANGE_RECORDS.md` | Planner/reporter | When scope or implementation changes require traceability |
| `RETROSPECTIVE.md` | Reporter/orchestrator | Sprint/project close |

These lifecycle docs may be required before final delivery, but they must not be confused with the documents required before implementation starts.

## G1 Documentary Readiness Gate checklist

A project cannot dispatch normal implementation work until this checklist is GREEN:

- [ ] `DOCUMENTATION_INDEX.md` exists and lists every G1 required document with status, owner, reviewer, and path.
- [ ] Every G1 required document exists in the canonical project artifact directory.
- [ ] Each required document is non-empty and contains project-specific content, not a placeholder.
- [ ] Each required document has been validated by the responsible role.
- [ ] Each required document has been independently reviewed or explicitly marked as Jean-approved.
- [ ] `TASK_GRAPH.md` is derived from the documents, not from UI state alone.
- [ ] `SPRINT_PLAN.md` and `TASK_GRAPH.md` agree on sequencing/dependencies.
- [ ] `ADRS.md` and `TECHNICAL_BLUEPRINT.md` agree on architecture boundaries.
- [ ] `QA_GATES.md` and `SECURITY_GATES.md` define the verification commands/evidence required for the project risk level.
- [ ] All required docs are indexed in `DOCUMENTATION_INDEX.md`.
- [ ] Project artifact directory has no uncommitted changes unless Jean explicitly authorized a commit waiver.
- [ ] Factory DB records a G1/documentary readiness gate result with evidence.
- [ ] Factory status/API exposes the per-document checklist to the UI.

## Runtime/code requirements

The implementation must update the Factory runtime so:

1. Required-document definitions are first-class and distinguish:
   - G1 blocking documents required before implementation.
   - Lifecycle/report documents generated during execution.
   - Optional PM projection documents such as Notion updates.
2. `status()` / Factory API exposes per-project document status, including:
   - file name
   - category (`g1_required`, `lifecycle`, `pm_projection`)
   - exists
   - indexed in `DOCUMENTATION_INDEX.md`
   - committed / git status
   - validated
   - reviewed
   - blocking status
   - owner/reviewer when known
3. Dispatch preflight blocks normal implementation only on G1 required-document failures, repository strategy failures, unresolved gates, or commit/index drift.
4. Missing Notion metadata/page is not a default implementation blocker and not a critical readiness blocker.
5. Notion remains visible and important as PM projection/reporting; missing/stale Notion produces a warning/reporting task unless project metadata explicitly marks Notion mandatory.
6. `DOCUMENTATION_INDEX.md` stale status should be detected before acceptance.
7. Existing stale artifacts in `factory-runtime-docs-notion-refactor` must be reconciled so task graph/tracker/index/reviews do not contradict Factory DB.

## Required regression tests

Add focused tests proving:

1. Docs ready + Notion missing => implementation dispatch is allowed by default.
2. Docs missing => implementation dispatch is blocked.
3. Docs exist but are missing from `DOCUMENTATION_INDEX.md` => blocked.
4. Required docs uncommitted => critical readiness/delivery fails.
5. Factory `status()` payload includes a `document_status` or equivalent field with required docs and per-file state.
6. Notion mandatory metadata changes behavior: when `notion_required=true`, missing Notion can block/report as configured.
7. Lifecycle docs such as `QA_REPORT.md` are not required before implementation unless configured for the current gate.
8. Final delivery fails if lifecycle reports required by later gates are missing/stale.

## Acceptance criteria

- Factory DB remains canonical for operational state.
- Repo Markdown docs remain canonical for engineering/project truth.
- Notion is important and visible, but not treated as source of truth.
- First implementation dispatch requires G1 Documentary Readiness Gate GREEN.
- Factory UI/API can show Jean the required-document checklist and current per-document status.
- Stale/contradictory docs in the current project are corrected or explicitly superseded with evidence.
- Focused pytest suite passes.
- A live Factory smoke confirms no implementation task is claimable before G1 docs are ready.
- CRM/Funnel Core remains frozen until this hotfix is GREEN and Jean explicitly approves CRM work.

## Non-goals

- Do not refactor CRM/Funnel Core in this hotfix.
- Do not downgrade Notion to irrelevant; keep it as a strong human PM/reporting layer.
- Do not solve by adding broad waivers.
- Do not create a detached project unless the current project cannot be reopened safely.
